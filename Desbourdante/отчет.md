# Отчет по заданию: Анализ и очистка данных конференции

В данном отчете проводится анализ данных о студенческой конференции (`student_conference_polluted.csv`) с использованием инструментов из библиотеки `desbordante`. Цель работы — проверить и обеспечить выполнение бизнес-правил с помощью **валидатора UCC** и **майнера FD**, обнаружить ключевые зависимости и выявить "почти верные" правила в наборе данных.

---

## 1. Проверка и исправление бизнес-правила (Уникальность 'Участник-Год')

**Цель:** Проверить бизнес-правило: "Один участник (уникальный `Email`) может выступить не более чем с одним докладом за год". В случае нарушения — удалить строки-артефакты и убедиться, что правило выполняется на очищенных данных.

### 1.1. Формулировка правила и метод проверки

Данное бизнес-правило эквивалентно требованию **уникальности комбинации столбцов (Unique Column Combination, UCC)** для `[Email, Year]`. Если в таблице найдется более одной строки с одинаковой парой значений в этих столбцах, правило нарушено.

Для проверки мы используем **валидатор UCC** (`aucc_verification`), который предназначен для поиска таких дубликатов.

### 1.2. Проверка на исходных ("загрязненных") данных

Была проведена проверка уникальности комбинации столбцов `Email` (индекс 1) и `Year` (индекс 6) на исходном файле `student_conference_polluted.csv`.

**Результат:**
Валидатор подтвердил, что правило нарушено, и обнаружил 2 кластера-нарушителя.

```
This set of attributes doesn't ensures uniqueness.
Clusters violating UCC:
found 2 clusters violating UCC:
Violating cluster 1:
    Speaker_Name             Email             Track             Paper_Title  Paper_Number       Status  Year

50  Howard Jason  howard@email.com  machine learning  Predictive Maintenance            35  participant  2024

        Speaker_Name              Email             Track                              Paper_Title  Paper_Number       Status  Year
94    Wheeler Joseph  wheeler@email.com  machine learning           Natural Language Understanding            95  participant  2024
100  Wheeler Richard  wheeler@email.com  machine learning  Advanced Natural Language Understanding           101  participant  2024
```

### 1.3. Анализ и устранение артефактов

-   **Анализ кластера 1:** Записи для `Howard Jason` являются полными дубликатами. Это классический пример строки-артефакта, которую необходимо удалить.
-   **Анализ кластера 2:** Для email `wheeler@email.com` в 2024 году зарегистрировано два разных участника (`Joseph Wheeler` и `Richard Wheeler`) с разными докладами. Это более серьезное нарушение целостности данных, прямо противоречащее бизнес-правилу.
-   **Устранение:** Для исправления данных артефакты (например, строка с индексом 50 и одна из строк для `wheeler@email.com`) были удалены, и данные сохранены в файл `student_conference_fixed.csv`.

### 1.4. Повторная проверка на очищенных данных

После удаления артефактов была проведена повторная проверка на файле `student_conference_fixed.csv`.

**Результат:**
```
This set of attributes ensures uniqueness.
```

**Вывод:** После очистки данных валидатор UCC подтвердил, что комбинация `[Email, Year]` стала уникальной для всей таблицы. Бизнес-правило теперь соблюдается.

## 2. Обнаружение ключевой зависимости (Майнинг FD)

**Цель:** Используя майнер FD с нулевой ошибкой (`error=0`), найти набор атрибутов `X`, который однозначно определяет все остальные атрибуты `Y` в таблице.

### 2.1. Выполнение майнера FD

На тех же данных был запущен майнер AFD для поиска строгих функциональных зависимостей.

**Результат (ключевые зависимости):**
```
[Paper_Number Year] -> Speaker_Name
[Paper_Number Year] -> Email
[Paper_Number Year] -> Track
[Paper_Number Year] -> Paper_Title
[Paper_Number Year] -> Status
```

### 2.2. Анализ и выводы

-   **Анализ:** Майнер обнаружил, что комбинация атрибутов `[Paper_Number, Year]` является левой частью для всех функциональных зависимостей, определяющих каждый из оставшихся атрибутов в таблице. Проверка этих FD с помощью верификатора подтвердила их выполнение:
    ```
    got [Paper_Number Year] -> Y, where Y -- some atribute. Lets check this rules:
    [Paper_Number, Year] -> Speaker_Name
    This FD holds
    [Paper_Number, Year] -> Email
    This FD holds
    ...
    ```
-   **Вывод:** Это означает, что пара `(Номер доклада, Год)` уникально идентифицирует любую запись в таблице. Таким образом, второе бизнес-правило и, по сути, **кандидатный ключ** данной таблицы — это **`[Paper_Number, Year]`**.

## 3. Поиск приближенных функциональных зависимостей

**Цель:** Найти правила, которые "почти" выполняются в данных, то есть имеют очень малое количество исключений.

### 3.1. Метод поиска

Был запущен майнер AFD в цикле с постепенным увеличением порога ошибки (`error`). Частота обнаружения каждой FD на разных уровнях ошибки показывает, насколько она "устойчива" и близка к выполнению.

**Результат:**
```
FDs close to hold:
99 count: [Paper_Number] -> Speaker_Name
99 count: [Paper_Number] -> Email
99 count: [Paper_Number] -> Paper_Title
99 count: [Paper_Title] -> Speaker_Name
99 count: [Paper_Title] -> Email
99 count: [Paper_Title] -> Paper_Number
99 count: [Email] -> Speaker_Name
99 count: [Email] -> Paper_Title
99 count: [Email] -> Paper_Number
99 count: [Speaker_Name] -> Email
```

### 3.2. Анализ и выводы

Наиболее устойчивые "почти верные" зависимости (встретились 99 раз из 100 итераций) и их интерпретация:

1.  **`[Paper_Number] -> Speaker_Name`**, **`[Paper_Number] -> Email`**, **`[Paper_Number] -> Paper_Title`**: Эти зависимости показывают, что номер доклада (`Paper_Number`) сам по себе почти однозначно определяет докладчика, его email и название доклада. Они не являются строгими (100% верными), потому что номера докладов могут быть переиспользованы в разные годы. Например, доклад №1 в 2023 году и доклад №1 в 2024 году — это разные доклады. Это наблюдение идеально подтверждает вывод из п.2 о том, что для полной уникальности необходим `Year`.

2.  **`[Email] -> Speaker_Name`**: Адрес электронной почты почти всегда однозначно определяет имя участника. Редкие исключения могут быть вызваны ошибками ввода данных или ситуациями, когда один email используется несколькими людьми (маловероятно в данном контексте).

3.  **`[Speaker_Name] -> Email`**: Имя участника почти всегда определяет его email. Это правило слабее предыдущего, так как могут существовать тезки (например, два разных "John Smith" с разными email).

**Общий вывод:** Анализ приближенных FD позволил выявить сильные, но неабсолютные связи в данных. Главный инсайт — атрибуты `Paper_Number`, `Email`, `Speaker_Name` и `Paper_Title` тесно связаны и почти взаимозаменяемы для идентификации доклада в рамках одного года, но для глобальной уникальности требуется добавление атрибута `Year`.
